# potential-data-playground

## Проект: модель скоринга кредитоспособности заемщика для банка

В качестве примера выбрана **градиентная бустинговая модель для бинарной классификации**: на вход она получает анкетные, транзакционные и поведенческие признаки, а на выходе возвращает **PD (probability of default)** — вероятность дефолта заемщика в заданном горизонте.

Ниже описан не код, а **процесс выпуска модели в production**: как команда работает по GitHub Flow, как оформляет релизы, как реагирует на инциденты с данными и как организует CI/CD.

---

## 1. Почему для этой команды подходит GitHub Flow

### Контекст команды
- **Размер команды:** 6–8 человек
  - 2 Data Scientist
  - 2 ML/Data Engineer
  - 1 Backend/MLOps Engineer
  - 1 QA/Analyst
  - 1 Product Owner / Risk Manager
- **Частота релизов:** небольшие поставки 1–2 раза в месяц, экстренные hotfix-релизы по необходимости.
- **Критичность системы:** высокая. Ошибка модели может привести к неверным кредитным решениям, финансовым потерям и регуляторным рискам.

### Почему именно GitHub Flow
GitHub Flow подходит здесь лучше всего, потому что:

1. **Есть одна защищённая production-ветка (`main`)**  
   Это упрощает контроль того, что именно считается боевой версией модели и связанных с ней артефактов.

2. **Каждое изменение идёт через короткоживущую feature-ветку и Pull Request**  
   Команда может независимо дорабатывать признаки, пайплайны, валидацию данных и документацию, не ломая стабильную версию.

3. **Процесс хорошо сочетается с частыми маленькими поставками**  
   Для ML-проекта безопаснее выпускать небольшие изменения: новый признак, фиксацию data leak, обновление порогов, чем копить крупный batch изменений.

4. **PR даёт обязательную точку контроля**  
   Для кредитного скоринга важны ревью кода, ревью данных, воспроизводимость обучения, проверка метрик, bias/fairness checks и согласование с risk/business stakeholder'ами.

5. **Релиз удобно привязывать к git tag и артефактам модели**  
   Тег версии фиксирует код, конфиг фичей, схему данных, отчёт по метрикам, параметры обучения и сам сериализованный артефакт модели.

### Ограничения и как их компенсировать
GitHub Flow прост, но для критичной банковской системы его нужно усилить процессом:
- protected branch rules для `main`;
- обязательные PR review;
- обязательные проверки CI;
- ручной approval перед выкладкой в production;
- обязательный rollback plan;
- хранение model card, training manifest и release notes рядом с релизом.

## 2. Описание релизного цикла

### Общие правила версионирования
Используем **SemVer**:
- **MAJOR (`v2.0.0`)** — несовместимое изменение контракта модели или её назначения;
- **MINOR (`v1.1.0`)** — обратно совместимое улучшение, например новый сильный признак;
- **PATCH (`v1.0.1`)** — исправление дефекта без изменения контракта.

Каждый релиз включает:
- git tag;
- артефакт модели;
- конфиг/схему признаков;
- зафиксированную версию train/inference pipeline;
- отчёт по качеству и стабильности;
- release notes;
- план отката.

### Выпуск первого релиза `v1.0.0`

#### Шаги
1. **Формулируем baseline**
   - согласуем целевую переменную;
   - фиксируем business KPI, ML-метрики и acceptance criteria;
   - фиксируем train/validation/test split и anti-leakage правила.

2. **Работаем в feature-ветках**
   - `feature/baseline-scoring-model`
   - `feature/data-validation-rules`
   - `feature/model-monitoring-contract`

3. **Открываем Pull Request**
   - прикладываем описание изменений;
   - прикладываем метрики на offline-валидации;
   - прикладываем перечень используемых признаков;
   - прикладываем evidence воспроизводимости (seed, dataset snapshot, config).

4. **CI проверяет PR**
   - стиль и статический анализ;
   - unit-тесты;
   - тесты feature engineering;
   - проверку схемы данных;
   - smoke train / smoke inference;
   - упаковку артефакта;
   - генерацию model report.

5. **Ревью и approval**
   - code review от инженера;
   - review от Data Scientist;
   - approval от risk/business owner;
   - проверка, что модель не нарушает регуляторные ограничения.

6. **Слияние в `main`**
   - после merge создаётся release candidate артефакт;
   - он выкатывается в staging / shadow mode.

7. **Staging / shadow validation**
   - сверяем онлайн- и офлайн-значения признаков;
   - проверяем latency, отказоустойчивость и корректность схемы ответа;
   - смотрим распределения score и долю отклонённых заявок.

8. **Релиз**
   - создаём tag `v1.0.0`;
   - публикуем changelog;
   - выкатываем модель в production;
   - включаем мониторинг drift, качества и бизнес-метрик.

### Что произойдёт при обнаружении data leak и как выпустить `v1.0.1`

#### Ситуация
После релиза команда обнаружила, что один из признаков использует информацию, недоступную на момент принятия кредитного решения, например агрегат, рассчитанный с заглядыванием в будущее.

#### Действия
1. **Останавливаем дальнейшее раскатывание**
   - блокируем новые feature-релизы;
   - открываем incident ticket.

2. **Оцениваем влияние**
   - определяем, в каких версиях был leak;
   - оцениваем, повлиял ли он только на offline-оценку или уже на production scoring;
   - готовим список затронутых решений и период времени.

3. **Создаём hotfix-ветку**
   - `hotfix/remove-leaky-feature`

4. **Исправляем дефект**
   - удаляем или пересчитываем leaky-признак;
   - обновляем правила data validation;
   - добавляем автоматическую anti-leakage проверку, чтобы ошибка не повторилась.

5. **Переобучаем модель**
   - на корректном датасете;
   - сравниваем с `v1.0.0`;
   - убеждаемся, что контракт inference не изменился.

6. **Ускоренное, но обязательное ревью**
   - DS + MLOps + risk owner;
   - при необходимости — formal sign-off из-за инцидента.

7. **Выпускаем patch-релиз**
   - merge в `main`;
   - tag `v1.0.1`;
   - публикуем changelog с описанием инцидента и исправления;
   - откатываем production на новую безопасную модель.

8. **Post-incident activities**
   - оформляем RCA;
   - обновляем чек-лист релиза;
   - расширяем тесты на leakage detection.

### Что произойдёт при добавлении нового мощного признака и как выпустить `v1.1.0`

#### Ситуация
Команда Data Science нашла новый сильный признак, например агрегат по стабильности денежных поступлений, который заметно улучшает ROC-AUC, PR-AUC и business uplift.

#### Действия
1. **Создаём feature-ветку**
   - `feature/add-income-stability-feature`

2. **Реализуем изменение**
   - добавляем новый признак в train pipeline;
   - обновляем feature store / схему данных;
   - обновляем inference pipeline;
   - документируем источник данных и SLA по доступности признака.

3. **Проверяем совместимость**
   - выход модели остаётся тем же: PD;
   - API/контракт для потребителей не меняется;
   - значит это **minor-релиз**, а не major.

4. **Запускаем расширенную валидацию**
   - сравнение с текущей production-моделью;
   - backtesting;
   - stability check по временным срезам;
   - fairness / bias checks;
   - оценка влияния на бизнес-порог и policy rules.

5. **PR, ревью и staging**
   - PR содержит метрики "было / стало";
   - в staging и shadow mode проверяем доступность нового признака и долю пропусков;
   - при необходимости запускаем canary deployment.

6. **Релиз**
   - merge в `main`;
   - tag `v1.1.0`;
   - обновляем changelog и model card;
   - поэтапно раскатываем модель в production.

### Что произойдёт при кардинальном изменении выхода модели и как выпустить `v2.0.0`

#### Ситуация
Бизнес хочет, чтобы модель не только выдавала вероятность дефолта, но и **объясняла основной фактор риска** или даже возвращала несколько выходов: `pd_score + top_reason`.

Это уже меняет продуктовый контракт, интеграции и логику использования результата.

#### Действия
1. **Запускаем отдельный discovery**
   - согласуем новый контракт ответа;
   - согласуем формат объяснений;
   - определяем downstream-системы, которые нужно адаптировать.

2. **Создаём эпик и набор feature-веток**
   - `feature/v2-output-contract`
   - `feature/v2-reason-codes`
   - `feature/v2-monitoring`

3. **Проектируем совместимость**
   - либо запускаем parallel run `v1` и `v2`;
   - либо добавляем versioned endpoint / versioned model topic.

4. **Проводим полную регрессию**
   - тестируем новый контракт;
   - обновляем интеграционные тесты;
   - обновляем документацию для потребителей;
   - обновляем мониторинг не только score, но и reason codes.

5. **Пилот и миграция**
   - сначала shadow;
   - затем ограниченный pilot на части трафика;
   - затем полное переключение клиентов на `v2`.

6. **Major release**
   - merge в `main`;
   - tag `v2.0.0`;
   - публикуем migration guide;
   - фиксируем срок поддержки `v1`.

## 4. Проектирование CI/CD Pipeline

### Что должно происходить после каждого коммита в feature-ветку
После каждого push в feature-ветку CI должен запускать стандартный конвейер:

1. **Checkout + dependency setup**
   - загрузка кода;
   - восстановление зависимостей;
   - подготовка изолированного окружения.

2. **Линтеры и статический анализ**
   - style/lint для Python/SQL/конфигов;
   - проверка типов;
   - базовые security checks.

3. **Unit-тесты**
   - тесты преобразования признаков;
   - тесты utility-функций;
   - тесты сериализации/десериализации модели.

4. **Data validation**
   - проверка схемы входных данных;
   - проверка пропусков и допустимых диапазонов;
   - anti-leakage checks;
   - проверка train/serve consistency.

5. **Smoke training**
   - короткое обучение на маленьком сэмпле;
   - проверка, что pipeline собирается и модель обучается без ошибок.

6. **Smoke inference**
   - прогон типовых заявок;
   - проверка формата ответа, диапазона score и стабильности признаков.

7. **Integration tests**
   - проверка взаимодействия с feature store / базой / API модели;
   - проверка контрактов между компонентами.

8. **Сборка артефакта**
   - упаковка модели;
   - сохранение конфига, схемы признаков и метаданных;
   - публикация временного артефакта для PR.

9. **Генерация отчётов**
   - offline metrics report;
   - drift baseline snapshot;
   - summary для Pull Request.

10. **PR gate**
   - merge разрешён только если все обязательные проверки зелёные.

### Что делает CD после merge в `main`
1. собирает release candidate;
2. публикует versioned артефакт;
3. выкатывает в staging;
4. запускает smoke / regression checks;
5. ждёт ручной approval;
6. выкатывает в production;
7. включает мониторинг и алерты;
8. позволяет быстро откатиться на предыдущий тег.

## 5. Чек-лист верификации (UV)

Перед любым релизом команда проходит следующий чек-лист:

- [ ] Целевая переменная и горизонт прогноза согласованы с бизнесом
- [ ] Обучающий, валидационный и тестовый наборы зафиксированы и воспроизводимы
- [ ] Проведена проверка на data leakage
- [ ] Все обязательные CI-проверки завершились успешно
- [ ] Метрики модели не хуже согласованного порога
- [ ] Сравнение с текущей production-версией задокументировано
- [ ] Проверены fairness / bias / stability показатели
- [ ] Inference-контракт и схема признаков зафиксированы
- [ ] Артефакт модели, конфиг и release notes сохранены
- [ ] Подготовлен rollback plan
- [ ] Staging/shadow/canary проверка выполнена успешно
- [ ] Бизнес-владелец и risk owner дали approval
- [ ] Настроены мониторинг качества, drift и технические алерты

## 6. Структура релизов и артефактов

Для каждой версии рекомендуется хранить:
- `model artifact` — сериализованная модель;
- `feature schema` — список и типы признаков;
- `training manifest` — источник данных, snapshot, seed, параметры;
- `metrics report` — ROC-AUC, PR-AUC, calibration, stability;
- `model card` — назначение, ограничения, риски;
- `release notes` — изменения относительно предыдущей версии.

Подробный пример истории релизов см. в файле [CHANGELOG.md](CHANGELOG.md).

## 7. FastAPI API для dummy scoring model

### Клиентский путь (User Story)
Кредитный аналитик в банке заполняет анкету клиента в CRM-системе. После нажатия кнопки **«Рассчитать скоринг»** CRM отправляет JSON с данными клиента в наш API и получает в ответ оценку риска. Результат отображается сотруднику как вспомогательная рекомендация при принятии решения.

### Функциональные требования
1. Сервис должен запускаться командой `uvicorn app.main:app`.
2. Сервис должен принимать `POST /predict` с JSON-объектом, содержащим поля:
   - `age` — целое число;
   - `income` — число с плавающей точкой;
   - `months_on_book` — целое число;
   - `credit_limit` — число с плавающей точкой.
3. Сервис должен возвращать JSON с полями:
   - `model_version` — версия загруженной dummy-модели;
   - `prediction` — строка с классом риска;
   - `score` — числовой скор модели.
4. Модель должна загружаться один раз при старте приложения через обработчик `@app.on_event("startup")`.
5. Если обязательные поля отсутствуют или имеют неверный тип, сервис должен возвращать `422 Unprocessable Entity`.
6. Сервис должен предоставлять endpoint `GET /health`, который всегда возвращает `{"status": "ok"}`.
7. Swagger-документация должна быть доступна по адресу `/docs`.

### Проектирование endpoint

**Метод:** `POST`  
**URL:** `/predict`

**Пример Request Body**

```json
{
  "age": 35,
  "income": 75000.0,
  "months_on_book": 24,
  "credit_limit": 15000.0
}
```

**Пример успешного ответа**

```json
{
  "model_version": "1.0-tabular",
  "prediction": "low_risk",
  "score": 0.8421
}
```

**Пример ответа с ошибкой**

```json
{
  "detail": [
    {
      "type": "missing",
      "loc": ["body", "months_on_book"],
      "msg": "Field required",
      "input": {
        "age": 35,
        "income": 75000.0,
        "credit_limit": 15000.0
      }
    }
  ]
}
```

### Локальный запуск
```bash
pip install -r requirements.txt
uvicorn app.main:app
```

### Тесты
```bash
pytest
```

## 9. Docker

### Сборка базового образа (Часть 1)

```bash
docker build -t ml-app:1.0 .
```

### Запуск контейнера и тестовый запрос

```bash
docker run -d -p 8000:8000 --name ml-app ml-app:1.0

# health-check
curl http://localhost:8000/health

# тестовый запрос
curl -X POST http://localhost:8000/predict \
  -H "Content-Type: application/json" \
  -d '{"age": 35, "income": 75000, "months_on_book": 24, "credit_limit": 15000}'

docker stop ml-app && docker rm ml-app
```

### Docker Compose

```bash
# Запуск
docker-compose up -d

# Остановка
docker-compose down
```

### Многоэтапная сборка (Часть 2)

```bash
docker build -f Dockerfile.multi -t ml-app:1.0-slim .
```

### Сравнение размеров образов

| Образ            | Размер  |
|------------------|---------|
| `ml-app:1.0`     | ~250 MB |
| `ml-app:1.0-slim` | ~220 MB |

> Точные значения зависят от окружения. Многоэтапная сборка уменьшает
> итоговый образ за счёт того, что в финальный слой не попадают кэши pip
> и промежуточные артефакты сборки.
